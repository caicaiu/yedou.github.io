# 为什么有集群

1. 并发量高，Redis性能会受到影响。一台远远不够处理

2. 存储。Redis 所有的数据都放在内存中，如果数据量大，很容易受到硬件的限制。

3. 可用性和安全。Redis宕机后，数据无法恢复

# 主从复制

redis的复制功能可以很好的实现数据库的读写分离，提高[服务器](https://www.yisu.com/)的负载能力。主数据库主要进行写操作，而从数据库负责读操作。

## 主从赋值同步原理

第一次复制

+ **Replication Id**：简称replid，是数据集的标记，id一致则说明是同一数据集。
+ **offset**：偏移量，随着repl_baklog中的数据增多而逐渐增大。slave完成同步时也会记录当前同步的offset。如果slave的offset小于master的offset，说明slave数据落后于master，需要更新。

> slave是从的时候，建立连接发送 replid和offset给master，如果不一致，就是第一次，slave把master的replid和offset保存到版本信息

# 哨兵

监听集群是否发生故障(master宕机)

1. 监控：不断检查您的master和slave是否工作

3. 通知：主发生故障，发送信息给服务端(java程序),更换写的数据库

## 哨兵原理

每隔1秒向集群的每个实例发送ping命令。

主观下线：发现某节点未在规定时间响应，则认为该实例主观下线。
客观下线：超过数量（quorum)的sentinel都认为该实例主观下线，则该实例客观下线。quorum值最好超过Sentinel实例数量的一半。

## 选取主结点的原理

1. slave与master断开的时长，超过(down-after-milliseconds*10)的slave会优先排除
2. 判断slave节点的slave-priority值，越小优先级越高，0永不为主
3. slave-prority一样，判断slave节点的offset值，越大说明数据越新，优先级越高
4. 判断slave节点的运行id大小，越小优先级越高。

## 选取主节点后的操作

1. sentinela给备选的slave节点发送**slaveof no one**命令，让slave成为master

2. sentinel给其它slave.发送**slaveof   192.168.200.130 7002**命令，让这些slave成为新masterl的从节点，同步数据
3. 最后sentinel将故障节点标记为slave,当故障节点恢复后会自动成为新的master的slave节点



## 客户端读写分离

Springl的RedisTemplate底层利用lettuce实现了节点的感知和自动切换。

1. 引入依赖
	
	```xml
	<dependency>
	    <groupId>org.springframework.boot</groupId>
	    <artifactId>spring-boot-starter-data-redis</artifactId>
	</dependency>
	```
	
	
	
2. 配置哨兵

	```yml
	spring:
	  redis:
	    sentinel:
	      master: mymaster
	      nodes:
	        - 192.168.200.130:27001
	        - 192.168.200.130:27002
	        - 192.168.200.130:27003
	```

3. 配置自动读写分离

	```java
	@Bean
	public LettuceClientConfigurationBuilderCustomizer clientConfigurationBuilderCustomizer(){
	    return clientConfigurationBuilder -> clientConfigurationBuilder.readFrom(ReadFrom.REPLICA_PREFERRED);
	}
	```

## 散列插槽

不同的keyRedis会计算哈希值，再取16384分配到不同的Redis上

Redis如何判断某个key应该在哪个实例？

- 将16384个插槽分配到不同的实例
- 根据key的有效部分计算哈希值，对16384取余
- 余数作为插槽，寻找插槽所在实例即可

如何将同一类数据固定的保存在同一个Redis实例？

- 这一类数据使用相同的有效部分，例如key都以{typeId}为前缀 访问的时候也要加上前缀

## 集群伸缩

1. 添加集群

	1. 创建一个文件夹：

		```sh
		mkdir 7004
		```


	2. 拷贝配置文件：

		```
		cp redis.conf /7004
		```

		

	3. 修改配置文件：

		```properties
		port 7004
		# 开启集群功能
		cluster-enabled yes
		# 集群的配置文件名称，不需要我们创建，由redis自己维护
		cluster-config-file /opt/7004/nodes.conf
		# 节点心跳失败的超时时间
		cluster-node-timeout 5000
		# 持久化文件存放目录
		dir /opt/7004
		# 绑定地址
		bind 0.0.0.0
		# 让redis后台运行
		daemonize yes
		# 注册的实例ip
		replica-announce-ip 192.168.200.130
		# 保护模式
		protected-mode no
		# 数据库数量
		databases 1
		# 日志
		logfile /opt/7004/run.log
		```

	4. 启动

		```
		redis-server 7004/redis.conf
		```

	5. 添加新节点

		```properties
		redis-cli --cluster add-node  192.168.200.130:7004 192.168.200.130:7001
		```

	6. 查看集群状态

		```
		redis-cli -p 7001 cluster nodes
		```

	7. 没有插槽这个实例是没有用的，所以分配插槽给当前实例,这里并指定范围，直接分配最下面的给新的

		```
		redis-cli --cluster reshard 192.168.200.130:7001
		分配3000
		谁来接收   7004他的id   通过查询集群状态获取
		从哪里获取  7001d的id   
		输入done
		输入yes
		```

	8. 删除一个结点

		+ 查看帮助命令

			```
			redis-cli --cluster help
			```

		+ 调整插槽

			```
			redis-cli reshard  192.168.3.15:6382 
			```

			

		+ 删除节点

			```
			redis-cli --cluster del-node 192.168.200.130:7004  19e0863fa235096d8c61395737a7073e95e62663
			```
